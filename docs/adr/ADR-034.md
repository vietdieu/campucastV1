# ADR-034: Interjection Before Interaction

## Status
Proposed (2026-07-04)

## Context
As CommuteCast moves into the **Conversational AI Era**, we need to define how the AI Radio Host communicates with the listener. Traditional chat interfaces (separate windows) break the immersion of a radio stream. The user requested an "Interactive radio host dialog" that explains things "without opening a separate chat".

## Decision
We will prioritize **Contextual Interjection** as the primary conversational pattern. Instead of a standalone chat agent, the AI Host is treated as a dynamic component of the broadcast timeline.

1.  **Dialogue as Timeline Beats**: All host dialogue (explanations, interjections) must be modeled as `Interjection` beats in the `ListeningSession`.
2.  **Narrative Shifting**: When a conversational interjection is triggered, the `InterjectionOrchestrator` must dynamically shift the remaining timeline to maintain continuity.
3.  **Situational Host**: Dialogue generation must be grounded in the current `TasteProfile` and `ResolvedPersona` to ensure the host sounds consistent with the rest of the show.

## Rationale
Radio is a linear, push-based medium. By treating dialogue as part of the timeline, we preserve the "stream" feel while providing the interactive depth the user desires. This approach avoids the cognitive load of switching between "listening mode" and "chat mode".

## Consequences
- **Positive**: High immersion, consistent persona, no UI clutter.
- **Negative**: Requires real-time timeline recalculation; Gemini API latency must be managed to avoid "dead air" before an interjection.
- **Mitigation**: Use skeletal interjection beats (pre-planned placeholders) while the model generates the final text.
