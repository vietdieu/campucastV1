# ADR-008: Pipeline Context Ownership

## Context
Orchestrating complex domain processes like briefing generation often leads to sprawling methods with deep parameter chains. Passing individual values between steps creates coupling and makes it difficult to add, remove, or modify steps without changing method signatures.

## Decision
We will enforce a strict "Pipeline Context Ownership" model for the briefing generation pipeline:
1. **Single Context Object**: Introduce a `PipelineContext` structure containing all operational state (e.g., raw RSS content, normalized articles, content fingerprints, generated summaries, generated entities, run identifiers, and execution status).
2. **Step Interface**: Every pipeline step must implement a uniform contract:
   ```typescript
   interface PipelineStep {
     execute(context: PipelineContext): Promise<Result<PipelineContext>>;
   }
   ```
3. **Immutability of Runs**: While the context is mutated sequentially by steps (or returns a new context), steps cannot bypass the context to share state. State propagates *exclusively* via the context object.

## Consequences
- Steps are completely decoupled from each other, knowing only the structure of the shared context.
- High testability; any step can be unit-tested in isolation by seeding a test context.
- Extremely simple step additions, removals, and re-orderings.
- Standardizes operational logging, audit tracking, and telemetry recording.
