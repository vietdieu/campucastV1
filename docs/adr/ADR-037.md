# ADR-037: Knowledge Snapshot First

## Status
Proposed (2026-07-04)

## Context
Our architecture relies on an offline-only learning loop (ADR-036). To maintain the stability and determinism of the CommuteCast Runtime, we must decouple the learning process (which produces new intelligence parameters) from the execution process (which consumes them).

## Decision
1. **Immutable Snapshots**: The Runtime component MUST NOT read directly from an active learning database or transient model training state. It operates exclusively on "Knowledge Snapshots" — immutable, versioned bundles of configuration and model parameters.
2. **Versioned Intelligence**: Every intelligence component (Recommendation Engine, Taste Graph, Persona, Editorial Policy) must be versioned within the Snapshot. The Runtime consumes a complete, consistent snapshot of these components for any given session.
3. **Decoupled Deployment**: Model training (offline) and Runtime deployment (configuration update) are decoupled. A new model version is only active when a new Snapshot is published and loaded by the Runtime.

## Rationale
This architecture mimics best practices in distributed systems (e.g., Kubernetes desired state, immutable infrastructure). It ensures:
- **Reproducibility**: Any session can be replayed or debugged by referencing the exact Snapshot ID.
- **Safety**: Rollbacks are trivial—reverting to a previous Snapshot ID returns the system to a known good state immediately.
- **Consistency**: The Runtime always operates with a coherent "worldview" (e.g., Taste Graph is consistent with Editorial Policy v3).

## Consequences
- **Positive**: Absolute Runtime stability, predictable behavior, auditability, effortless rollbacks.
- **Negative**: Adds operational overhead of snapshot management and versioning.
- **Mitigation**: Automated Snapshot Publishing pipeline as part of the Offline Learning Platform.
