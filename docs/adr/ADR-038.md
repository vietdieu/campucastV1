# ADR-038: Knowledge Registry

## Status
Proposed (2026-07-04)

## Context
Following the establishment of ADR-035 (Observe Before Learn), ADR-036 (Learning Happens Offline), and ADR-037 (Knowledge Snapshot First), we have successfully decoupled the learning process from the real-time execution engine (Runtime). To safely manage, audit, and deploy the immutable versioned snapshots produced by the Offline Learning Platform, we need a centralized authority that maintains version histories, tracks artifact origins, and guarantees 100% rollback capabilities.

## Decision
1. **The Knowledge Registry**: We introduce a dedicated `Knowledge Registry` as the single source of truth for all learned model snapshots, configuration states, and personalization parameters (such as Taste Profiles, Editorial Policies, Persona configurations, and Recommendation Weights).
2. **Immutability & Versioning (No Overwrite)**:
   - All registered artifacts MUST be assigned a unique, semantic, and sequential version ID (e.g., `snapshot-v1.2.3`).
   - Artifacts once written are strictly **immutable**. Under no circumstances can an existing snapshot version be updated, edited, or overwritten.
3. **Provenance & Lineage Tracking**: Every snapshot in the registry must contain a metadata manifest detailing:
   - **`parentSnapshotId`**: Pointing to the previous stable baseline to track lineage.
   - **`trainingDataset`**: Identifiers or time ranges of features extracted from the Learning Event Store.
   - **`validationReport`**: Programmatic scores from the validation suite proving that the snapshot meets our safety metrics (e.g., editorial score stability, diversity rules, no regression).
   - **`rollbackTargetId`**: Pre-validated backup version to fallback to in case of failure.
4. **Instant Rollback**: The runtime always loads a specific snapshot version ID. Reverting the system to a previous state is achieved by updating the desired active ID in the Registry, taking effect in < 1 minute with zero downtime or manual rebuilds.

## Rationale
Real-world systems with learning loops suffer from "drift" and "non-deterministic degradations." Overwriting models or serving dynamic weights directly makes auditing extremely difficult. By keeping a strict history registry, we can trace exactly why a specific recommendation was made, verify model improvements before production rollouts, and guarantee zero risk of catastrophic failures through 1-click rollbacks.

## Consequences
- **Positive**: Complete auditability, guaranteed rollback capability, clear system provenance, safe A/B testing baseline.
- **Negative**: Increased storage requirements for historical snapshot logs (mitigated by pruning un-deployed intermediary snapshots).
- **Mitigation**: Standardized registry schemas and automated metadata generation to keep tracking overhead minimal.
