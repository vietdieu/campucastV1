# ADR-014: User Memory as Source of Personalization

## Context
As CommuteCast evolves towards a personalized AI DJ experience, we need a robust mechanism to store and retrieve user preferences, listening history, and feedback. If we mix personalization logic (Recommendation Engine) directly with data storage, the system will become tightly coupled and difficult to test or evolve.

## Decision
We establish the **AI Memory Platform** as the singular "Source of Truth" for all user personalization data, adhering strictly to the following architectural constraints:
1. **Separation of Concerns**: The Memory Platform only records events (e.g., `BriefingPlayed`, `SegmentSkipped`, `Favorite`) and state (`UserAggregate`, `InterestGraph`). It does **not** make decisions about what content to recommend.
2. **Domain-Driven Design**: Personalization data is encapsulated in the `UserAggregate`. Repositories (`MemoryRepository`) abstract the underlying persistence (e.g., PostgreSQL, In-Memory).
3. **Event-Based History**: We store atomic events for listening history and feedback rather than just mutating state. This allows future Recommendation Engines and Prompt Builders to replay or analyze history chronologically.
4. **No LLM/Recommendation Leakage**: The Memory Platform has no knowledge of LLMs, prompts, or recommendation algorithms.

## Consequences
- The Recommendation Engine (Sprint #009) can be safely built on top of this clean data layer.
- Changing recommendation algorithms won't require database schema migrations.
- Personalization logic is fully testable via in-memory repositories.
- Provides a clear boundary: Memory = Data Storage/Retrieval; Recommendation = Compute/Decision.
