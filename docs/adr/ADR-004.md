# ADR-004: Repository Pattern

## Context
As CommuteCast grows, the data access logic is tightly coupled with infrastructure (DB/Storage), making testing and swapping storage technologies (e.g., PostgreSQL, Redis) difficult.

## Decision
We will implement the Repository Pattern to abstract data access.
1. Define repository interfaces in the `domain` layer.
2. Implement infrastructure-specific adapters in the `infrastructure` layer.
3. Domain services will depend only on the interfaces.

## Consequences
- Business logic is completely decoupled from storage implementation.
- Testing becomes easier (mock repositories).
- We can easily swap out `MemoryRepository` for a `PostgreSQLRepository` in the future.
