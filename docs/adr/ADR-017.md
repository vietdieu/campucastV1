# ADR-017: Product Intelligence First (Candidate Enrichment)

## Context
Our ingestion pipeline currently produces `Candidate` objects that are mostly metadata-lite (freshness, feed score, confidence). To build a Spotify-like Recommendation Engine, we need to rank content based on semantic alignment with the user's Interest Vector. If the Recommendation Engine itself has to perform NLP, topic extraction, and sentiment analysis at ranking time, it will be slow and complex.

## Decision
We establish the **Candidate Intelligence Platform** as a mandatory enrichment gate before recommendation.
1. **Semantic Enrichment**: Every `Candidate` must be enriched with a `CandidateIntelligence` payload containing topics, keywords, named entities, and sentiment.
2. **Multi-dimensional Scoring Signals**: Add urgency, importance, and novelty scores to the intelligence metadata.
3. **Reading Complexity**: Calculate reading difficulty and estimated narrative duration to better fit the user's current `Context` (e.g., shorter stories for "Fast Commute").
4. **Decoupled Enrichment**: Enrichment can happen at ingestion time or on-demand, but must be available to the ranker as a static intelligence DTO.

## Consequences
- The Recommendation Engine becomes a pure mathematical ranker (Score = Vector A dot Vector B).
- We can change the "Intelligence" logic (e.g., switching from keyword extraction to LLM-based topic mapping) without impacting the ranking algorithm.
- Downstream AI DJ components can use these semantic tags to generate contextual transitions (e.g., "Switching to some more urgent tech news...").
