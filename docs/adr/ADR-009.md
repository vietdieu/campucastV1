# ADR-009: Deterministic Step Contract

## Context
Non-deterministic behavior in pipeline processing (e.g., direct calls to `Date.now()`, `Math.random()`, or inline UUID generators inside individual steps) makes debugging, telemetry replay, and unit testing highly complex. A robust pipeline must be reproducible and deterministic.

## Decision
We enforce a "Deterministic Step Contract" for all briefing generation pipeline steps:
1. **No Side-Effect Functions**: Individual steps are strictly forbidden from obtaining system state directly (e.g., retrieving system time, generating random values, or calling external system parameters inline).
2. **Context-Driven Parameters**: All non-deterministic variables (e.g., `runId`, processing `timestamp`, `schemaVersion`, `promptVersion`) must be injected into the `PipelineContext` at the pipeline's initialization phase.
3. **Pure Logic Steps**: If a step requires a timestamp, unique ID, or random seed, it must read it directly from the initialized properties inside the `PipelineContext`.

## Consequences
- The pipeline becomes completely deterministic for a given input and starting context.
- We can replay failed pipeline runs precisely by recreating the original input data and initial context state.
- Highly predictable and reliable unit tests, removing flaky behaviors or time-of-day dependent failures.
