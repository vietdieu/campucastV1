# ADR-015: Runtime Orchestration First

## Context
As CommuteCast grows, we have accumulated numerous independent modules: Feed Intelligence, LLM Platform, Memory Platform, Audio Manifest, and TTS Pipeline. If the UI or the Audio Player directly accesses these services to coordinate playback, caching, and state projection, the frontend becomes a tightly coupled monolith. Spotify's seamless AI DJ experience stems from a robust orchestration runtime, not just a TTS pipeline.

## Decision
We establish the **Runtime Orchestration Platform** as the sole coordinator and "brain" of the client application.
1. **Event-Driven Coordination**: All cross-module communication must happen via the `RuntimeEventBus` (e.g., `LanguageChanged` triggers manifest rebuilds and prefetching, without direct module coupling).
2. **Runtime Context**: A centralized `RuntimeContext` holds the unified state of the user, briefing, memory, and environment.
3. **Projection Engine**: The UI never consumes raw briefings or manifests. It consumes "Projections" (e.g., EN, VI, Bilingual) generated by the `ProjectionEngine`, allowing the UI to remain a pure view layer.
4. **Playback Scheduling**: A dedicated `PlaybackScheduler` manages segment sequencing and prefetching, removing this complexity from the raw PCM players.

## Consequences
- The UI becomes extremely thin, acting only as a view for the `RuntimeContext` projections.
- Module coupling is drastically reduced; features can be developed independently and wired together via events.
- Caching, telemetry, and prefetching can be centrally coordinated, ensuring gapless audio playback and accurate performance tracking.
