# ADR-036: Learning Happens Offline

## Status
Proposed (2026-07-04)

## Context
Following ADR-035 (Observe Before Learn), we have established a robust evaluation layer that scores performance without modifying system behavior. To continue scaling CommuteCast as a self-improving AI Radio platform, we must define the mechanism by which the observations collected by our evaluation layer translate into system improvements (learning).

## Decision
1. **Offline-Only Learning Loop**: Runtime components are strictly forbidden from modifying intelligence parameters (recommendation weights, persona traits, taste graph) in real-time.
2. **The Offline Learning Platform**: 
   - All evaluation data and performance metrics are ingested by an asynchronous `Offline Learning Job`.
   - This job performs feature extraction, trend analysis, and model parameter tuning.
   - The output is a new, validated set of model parameters (a new "checkpoint").
3. **Validation & Deployment**: Before any new model checkpoint is live, it must pass an automated validation suite (Regression Tests, Diversity/Novelty Constraints) and receive human/product-manager sign-off.

## Rationale
Real-time "online learning" introduces high risks of intelligence debt and non-deterministic behavior that is difficult to audit or debug. By enforcing an offline-only loop, we ensure that every system update is reproducible, validated, and aligned with overall Product KPIs.

## Consequences
- **Positive**: High system stability, easy rollbacks, auditability of model changes.
- **Negative**: Learning feedback cycle is no longer instantaneous; introduces operational overhead for model deployment.
- **Mitigation**: Automated pipeline for feature extraction and model validation to keep the offline cycle fast.
